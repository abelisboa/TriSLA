## Canonical Governance Reference

For the cross-module governance contract, use [`docs/modules/governance.md`](governance.md). Portal Backend is the **Governance Metadata Authority**: it derives and propagates `governance_registration_*` fields from BC-NSSMF evidence and persists them through SEM-CSMF.

## Official Identity

Portal Backend is the TriSLA operational entry point for frontend-facing API
traffic. Its official responsibility is:

- Ingress
- Orchestration relay
- Response aggregation
- Metadata propagation
- Frontend backend API layer

Portal Backend does not decide SLA.

Portal Backend does not execute admission logic.

Portal Backend does not produce governance.

Portal Backend does not produce runtime assurance.

Those responsibilities are delegated to the authoritative modules documented in
this page. Portal Backend may preserve local fallback paths for operational
continuity, but fallback code does not make Portal Backend the canonical
authority for admission, governance, lifecycle, or runtime assurance.

## Runtime Architecture Truth

Canonical runtime flow:

```text
Frontend
|
Portal Backend
|
SEM-CSMF
|
Decision Engine
|
Portal Backend
|
BC-NSSMF
|
SLA-Agent
|
Frontend
```

Decision Engine and ML-NSMF are reached indirectly through SEM-CSMF in the
canonical admission path. Portal Backend is not the Decision Engine and must not
be documented as if it calls the Decision Engine evaluation endpoint directly.

## Admission Runtime Truth

### Active Endpoint

```text
POST /api/v1/sla/submit
```

Canonical admission flow:

```text
Frontend
|
Portal Backend
|
SEM-CSMF
|
Decision Engine (internal orchestration)
|
SEM-CSMF
|
Portal Backend
|
Frontend
```

Portal Backend validates and normalizes the incoming API payload, collects the
submission-time telemetry snapshot, builds the operational request envelope,
forwards it to SEM-CSMF, receives the SEM-CSMF/Decision Engine result, and
returns an aggregated response to the frontend.

Portal Backend != Decision Engine.

Portal Backend does not call `/evaluate` as the canonical admission path.

Admission decision authority belongs to Decision Engine. Portal Backend only
propagates the resulting decision metadata and response fields.

## Governance Runtime Truth

### Active Authority

```text
BC-NSSMF
```

BC-NSSMF produces immutable governance evidence:

```text
tx_hash
block_number
bc_status
```

### Active Portal Metadata

Portal Backend derives, normalizes, and propagates governance registration
metadata from the BC-NSSMF response and upstream metadata:

```text
governance_registration_status
governance_registration_tx_hash
governance_registration_block_number
governance_registration_fallback
```

### Active Persistence Relay

Portal Backend persists governance metadata back to SEM-CSMF through:

```text
PATCH /api/v1/intents/{intent_id}/governance-metadata
```

Portal Backend = Governance Metadata Relay.

Portal Backend does not generate governance evidence. BC-NSSMF generates the
blockchain registration evidence, while Decision Engine remains the semantic
governance authority for decision metadata. This alignment preserves the 10G.4
and 10G.8 governance documentation expectations.

## Runtime Assurance Runtime Truth

### Active Endpoint

```text
POST /api/v1/sla/revalidate-telemetry
```

Canonical reassessment flow:

```text
Portal Backend
|
SLA-Agent
|
Portal Backend
```

Runtime assurance fields returned through Portal Backend are propagated data:

```text
runtime_assurance
metric_explainability
domain_compliance
violations
warnings
recommendations
drift
bottleneck
```

These fields are not generated by Portal Backend in the canonical runtime path.
SLA-Agent is the runtime assurance and reassessment authority. Portal Backend
may keep local fallback behavior for continuity when delegation is disabled or
unavailable, but the fallback is not the authoritative runtime model.

## Status Runtime Truth

### Active Endpoint

```text
GET /api/v1/sla/status/{sla_id}
```

Status is recomposed by aggregating:

- SEM-CSMF intent state
- telemetry snapshots and live telemetry collection
- SLA-Agent runtime assurance data when available
- governance metadata persisted through SEM-CSMF

Portal Backend exposes the recomposed status to the frontend. It does not become
the lifecycle, governance, or runtime assurance authority by exposing this
status view.

## Persistence Runtime Truth

| Data area | Runtime truth |
| --- | --- |
| Intent State | ACTIVE. Canonical intent state is read from and persisted through SEM-CSMF. |
| Governance Metadata | ACTIVE. Portal Backend relays normalized governance metadata to SEM-CSMF with `PATCH /api/v1/intents/{intent_id}/governance-metadata`. |
| Runtime Status | ACTIVE. Status is recomposed from SEM-CSMF, telemetry, SLA-Agent data, and governance metadata. |
| Cache | CONDITIONAL. Cache behavior is configuration/runtime dependent and must not be documented as the canonical persistence layer. |

Portal Backend has no canonical primary database authority for SLA admission,
governance, or runtime assurance. Optional telemetry CSV export and fallback
paths are operational support mechanisms, not the authoritative state store.

## Explainability Runtime Truth

### Active Propagated Fields

```text
decision_evidence
metric_explainability
domain_compliance
```

Portal Backend aggregates explainability data.

Portal Backend does not generate explainability authority.

Decision evidence originates from the admission pipeline. Runtime and metric
explainability are propagated from runtime assurance data, especially SLA-Agent
outputs and SEM-CSMF metadata. Portal Backend may flatten or normalize fields so
the frontend receives a stable API contract.

## Integrations

| Integration | Classification | Runtime role |
| --- | --- | --- |
| Frontend | ACTIVE | Sends SLA requests and consumes aggregated responses/status. |
| SEM-CSMF | ACTIVE | Canonical semantic/admission entry used by Portal Backend. |
| BC-NSSMF | ACTIVE | Produces blockchain governance evidence after accepted admission. |
| SLA-Agent | ACTIVE | Executes runtime reassessment and runtime assurance. |
| Decision Engine | INDIRECT | Admission decision authority orchestrated internally through SEM-CSMF. |
| ML-NSMF | INDIRECT | Prediction/ML role reached through the SEM-CSMF/Decision Engine path. |
| NASP Adapter | CONDITIONAL | Instantiation path used when orchestration is required and configured. |

## Implementation and Evidence Alignment

This document aligns Portal Backend with the frozen post-refactor baseline:

- Portal Backend is ingress, relay/delegator, response aggregator, and metadata
  propagator.
- Decision Engine remains the admission and semantic decision authority.
- BC-NSSMF remains the immutable governance evidence producer.
- SLA-Agent remains the runtime assurance and reassessment authority.
- Only `main.py` mounted routers are documented as active public APIs.
- Phase 47, Results Freeze, Runtime Freeze, 10G.4, and 10G.8 evidence remain
  preserved because this page changes documentation only, not runtime behavior.
