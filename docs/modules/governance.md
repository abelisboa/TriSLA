## Governance Runtime Truth

Canonical runtime flow:

```text
Portal Backend
|
POST /api/v1/register-sla
|
BC-NSSMF
|
SLAContract.sol
|
Besu
|
tx_hash
block_number
bc_status
|
Portal Backend
|
PATCH governance metadata
|
SEM-CSMF
|
Portal Frontend
```

Governance in TriSLA is not a single service-owned object. It is a cross-module
runtime chain where each module owns a bounded responsibility.

## Official Identity

```text
BC-NSSMF = On-Chain Evidence Authority
```

```text
Portal Backend = Governance Metadata Authority
```

```text
SEM-CSMF = Governance Metadata Persistence Layer
```

```text
Portal Frontend = Governance Visualization Layer
```

BC-NSSMF produces blockchain evidence. Portal Backend composes and normalizes
governance registration metadata. SEM-CSMF persists that metadata. Portal
Frontend renders it.

## Blockchain Runtime Truth

### Active Contract

```text
SLAContract.sol
```

Classification:

```text
ACTIVE
```

### Produced Evidence

```text
tx_hash
block_number
bc_status
```

Generated by:

```text
BC-NSSMF + Besu
```

The active on-chain path is `POST /api/v1/register-sla` in BC-NSSMF, which
submits `SLAContract.registerSLA()` to Hyperledger Besu and returns transaction
receipt fields to Portal Backend.

Penalty, compensation, and registry contract narratives must not be documented

## Governance Metadata Runtime Truth

### Produced Metadata

```text
governance_registration_status
governance_registration_tx_hash
governance_registration_block_number
governance_registration_fallback
```

Produced by:

```text
Portal Backend
```

Portal Backend derives these fields from BC-NSSMF evidence, admission outcome,
and available upstream metadata.

### Persisted Metadata

```text
PATCH
/api/v1/intents/{intent_id}/governance-metadata
```

Persisted by:

```text
SEM-CSMF
```

SEM-CSMF merges allowed governance keys into intent `extra_metadata`. It does
not generate the blockchain evidence and does not execute governance logic.

## Governance Propagation Runtime Truth

```text
BC-NSSMF
|
Portal Backend
|
SEM-CSMF
|
Portal Frontend
```

This propagation path implements the 10G.4 and 10G.8 governance expectations:
metadata is patched into SEM-CSMF and rendered by Portal Frontend.


### `lifecycle_lineage_id`

Classification:

```text
NOT WIRED
```

### `lineage`

Classification:

```text
PARTIAL
```

### `governance_event`

Classification:

```text
PARTIAL
```

Lineage is not the authoritative runtime path. The BC-NSSMF helper module can
build lineage-style blocks, but it is not imported by the active `register-sla`
handler. Portal Frontend currently exposes partial lineage through governance
event fields, lifecycle timeline, auditability metadata, and technical payloads.

## Decision Engine Governance Truth

| Item | Classification | Runtime note |
| --- | --- | --- |
| `decision_evidence` | ACTIVE | Quantitative evidence metadata attached to decisions |
| `governance_event` | PARTIAL | Metadata shape exists, but runtime authority enrichment is not fully wired |
| `lifecycle_authority.py` | NOT WIRED | Implemented helper, not authoritative runtime path |

Decision Engine is the admission decision authority. It must not be documented
as the active runtime producer of end-to-end governance registration metadata.
